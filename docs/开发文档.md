## 整体架构

### 技术选型

| 层级 | 选型 | 理由 |
| --- | --- | --- |
| 框架 | React 18 + TypeScript | 组件化、类型安全、生态成熟 |
| 构建 | Vite 5 | 启动快、HMR 流畅、配置极简 |
| 路由 | React Router 6 (HashRouter) | 静态部署友好,无需服务器配置 rewrite |
| 样式 | 原生 CSS + CSS 变量 | 无运行时,主题切换易扩展 |
| 第三方库 | 仅 `qrcode` | 二维码算法复杂,其余功能用浏览器原生 API |

### 路由策略:HashRouter vs BrowserRouter

选用 [HashRouter](../src/main.tsx#L8),URL 形如 `index.html#/json`。
- **优点**:静态服务器(Nginx、GitHub Pages、对象存储)无需配置 fallback,刷新页面不会 404
- **代价**:URL 多了 `#`,SEO 不友好(对工具站无所谓)

如果部署在自有 Nginx 且配好了 `try_files`,可换成 `BrowserRouter`。

### 目录约定

```
src/
├── main.tsx          # 入口:挂载 React、套 Router
├── App.tsx           # 路由表 + 整体壳子
├── components/       # 跨页面复用组件
├── pages/            # 一个工具一个文件,自包含
└── styles/global.css # 全局样式 + CSS 变量
```

每个工具页是**自包含**的:状态、转换逻辑、UI 都在同一文件里,方便单独阅读、删除或迁移。没有提前抽象「工具基类」或「公共表单」之类的,因为工具间形态差异大,过早抽象反而绑手绑脚。

## 1. JSON 格式化 / 预览

📄 文件:[JsonTool.tsx](../src/pages/JsonTool.tsx)

### 核心原理

#### 1.1 解析与校验

直接使用浏览器原生 `JSON.parse`,它的语法错误信息已经足够友好:

```ts
try {
  const obj = JSON.parse(input);
} catch (e) {
  setError((e as Error).message);
  // 形如 "Unexpected token } in JSON at position 42"
}
```

校验逻辑放在 `useEffect` 而非 `useMemo`(后者在严格模式下两次执行 + 渲染期 setState 会触发警告)。

#### 1.2 格式化与压缩

`JSON.stringify` 第三个参数是缩进:

```ts
JSON.stringify(obj, null, 2)  // 格式化,2 空格缩进
JSON.stringify(obj)           // 压缩,无空白
```

#### 1.3 转义 / 反转义

「转义」即把字符串再走一次 `JSON.stringify`,会自动加引号并转义内部的 `"` `\n` 等:

```ts
JSON.stringify('他说: "hi"\n')
// → "\"他说: \\\"hi\\\"\\n\""
```

「反转义」反过来用 `JSON.parse`,前提是输入本身是一个合法的 JSON 字符串(以 `"` 开头结尾)。

#### 1.4 语法高亮

用一个**正则一次性匹配** key、string、number、boolean、null、bracket,然后回调中按捕获组判断类型并包 `<span>`。这是 web 上经典做法,无需 AST,性能足够:

```ts
/("(?:\\.|[^"\\])*"\s*:)|("(?:\\.|[^"\\])*")|\b(true|false)\b|\b(null)\b|(-?\d+(?:\.\d+)?(?:[eE][+-]?\d+)?)|([{}\[\],])/g
```

> 关键:**先转义 HTML**(`&` `<` `>`),再做高亮替换,否则用户输入的 `<script>` 字面量会注入到 DOM。

输出用 `dangerouslySetInnerHTML` 渲染。这里安全因为:
- HTML 转义在前
- 替换只插入 `<span class="...">`,标签和类名都是写死的常量

#### 1.5 复制到剪贴板

```ts
await navigator.clipboard.writeText(input);
```

> Clipboard API 仅在 **HTTPS 或 localhost** 下可用,http 环境会静默失败。

### 可扩展方向

- **JSON Path 提取**:输入 `$.user.name` 提取字段
- **JSON ↔ YAML / TOML 互转**:引入 `js-yaml` 等
- **Schema 校验**:用 `ajv` 校验 JSON Schema
- **大文件优化**:5MB+ 的 JSON 高亮会卡,可换成虚拟滚动 + 分块高亮

## 2. 图片压缩

📄 文件:[ImageCompressor.tsx](../src/pages/ImageCompressor.tsx)

### 核心原理

整个压缩流程只用了**浏览器原生 Canvas API**,无需后端、无需第三方库,数据完全本地:

```
File → ObjectURL → Image → Canvas.drawImage → Canvas.toBlob → 新 ObjectURL
                                                  ↑
                                            质量参数 0~1
```

### 关键步骤

#### 2.1 文件读取(File / Blob → Image)

```ts
const url = URL.createObjectURL(file);  // 不读完整内容,只生成引用
const img = new Image();
img.src = url;
await new Promise(r => img.onload = r);
```

`URL.createObjectURL` 比 `FileReader.readAsDataURL` 更高效:不做 base64 编码,直接给一个内存引用。**用完应 `URL.revokeObjectURL` 释放**(本项目为简化未做,长期使用可优化)。

#### 2.2 缩放(等比例)

```ts
const ratio = Math.min(1, maxWidth / img.naturalWidth);
const outWidth = Math.round(img.naturalWidth * ratio);
const outHeight = Math.round(img.naturalHeight * ratio);
```

`Math.min(1, ...)` 保证不放大,只缩小。

#### 2.3 重绘到 Canvas

```ts
const canvas = document.createElement('canvas');
canvas.width = outWidth;
canvas.height = outHeight;
canvas.getContext('2d').drawImage(img, 0, 0, outWidth, outHeight);
```

`drawImage` 自带双线性插值,缩放后视觉质量基本满足日常需求。如需更高质量,可分多步缩放(每次缩 1/2)。

#### 2.4 编码输出

```ts
canvas.toBlob(blob => {...}, 'image/jpeg', 0.7);
//                            ↑ 输出格式    ↑ 质量 0~1(仅有损格式有效)
```

| 格式 | 质量参数 | 透明 | 说明 |
| --- | --- | --- | --- |
| `image/jpeg` | 有效 | ❌ | 通用,压缩率高 |
| `image/webp` | 有效 | ✅ | 体积更小,新浏览器支持 |
| `image/png` | 忽略 | ✅ | 无损,大小取决于 Canvas 像素数(仅靠尺寸压缩) |

> **注意**:PNG 是无损格式,质量参数无效,所以 UI 中切到 PNG 时质量滑块会禁用。PNG 想压更小,得靠尺寸缩小或用专业算法(如 [pngquant](https://pngquant.org/) 做调色板量化)。

### EXIF 与方向问题

iPhone 拍的照片有 EXIF 旋转信息,直接 `drawImage` 后**会丢失方向**(出现倒置)。完整方案:
1. 用 `exifr` 读 EXIF orientation
2. 根据值在 Canvas 上做对应的 `rotate / translate`

本工具未处理(避免引入依赖)。如果是面向手机用户的产品,务必加上。

### 可扩展方向

- **批量压缩**:循环处理 + 用 `JSZip` 打包下载
- **WebWorker 化**:大图压缩不阻塞主线程,用 `OffscreenCanvas` 在 Worker 里跑
- **二次压缩防抖**:质量滑动时实时预览(节流到 300ms)

## 3. 二维码生成

📄 文件:[QrCodeTool.tsx](../src/pages/QrCodeTool.tsx)

### 为什么用 `qrcode` 库

QR Code 编码涉及:
1. 数据分段(数字、字母数字、字节、汉字四种模式)
2. Reed-Solomon 纠错码生成
3. 矩阵布局 + 掩码选择(8 种掩码取最优)
4. 格式信息编码

完整实现约 2000 行代码,自己写得不偿失。`qrcode` 是社区最成熟的实现,体积约 50KB,直接用。

### 关键 API

```ts
QRCode.toCanvas(canvasEl, text, options, callback)
```

参数:

```ts
{
  width: 280,                  // 像素尺寸
  margin: 2,                   // 边缘空白模块数(标准要求 ≥ 4,这里小一些更紧凑)
  errorCorrectionLevel: 'M',   // L/M/Q/H,纠错率 7%/15%/25%/30%
  color: { dark: '#000', light: '#fff' },
}
```

### 纠错等级怎么选

| 等级 | 纠错率 | 适用场景 |
| --- | --- | --- |
| L | 7% | 干净屏幕显示 |
| M | 15% | 默认,纸面打印 |
| Q | 25% | 户外、磨损环境 |
| H | 30% | 中心有 Logo 遮挡(扣掉部分仍可识别) |

**等级越高 → 二维码模块越多 → 同样像素下扫描越难**。实践:能用 M 别用 H。

### 颜色对比度

`color.dark` 与 `color.light` 必须有**足够对比度**,否则扫不出。建议保持深色 + 浅色搭配,慎用「同色系装饰」。

### 下载实现

直接把 Canvas 转 dataURL 触发 `<a download>`:

```ts
const url = canvasRef.current.toDataURL('image/png');
const a = document.createElement('a');
a.href = url;
a.download = 'qrcode.png';
a.click();
```

dataURL 在大尺寸时(>4MB)有性能问题,本场景二维码很小不需要担心。如果输出大图,改用 `canvas.toBlob` + `URL.createObjectURL` 更高效。

### 可扩展方向

- **中心 Logo**:在 Canvas 上叠加图片,搭配 H 级纠错
- **二维码识别**:用 `jsqr` 解码上传的图片
- **批量生成 + 打包**:列表导入 → 批量生成 → ZIP 下载

## 4. Base64 编解码

📄 文件:[Base64Tool.tsx](../src/pages/Base64Tool.tsx)

### 为什么不能直接用 `btoa` / `atob`

浏览器内置的 `btoa(str)` **只能处理 Latin-1**(0~255)。中文字符会报错:

```ts
btoa('你好')
// Uncaught DOMException: The string to be encoded contains characters outside of the Latin1 range.
```

### 正确的 UTF-8 安全方案

先用 `TextEncoder` 转成 UTF-8 字节,再把字节当成 Latin-1 字符串喂给 `btoa`:

```ts
function utf8Encode(str: string): string {
  const bytes = new TextEncoder().encode(str);   // → Uint8Array (UTF-8 字节序列)
  let binary = '';
  bytes.forEach(b => binary += String.fromCharCode(b));  // 字节 → "Latin-1 字符"
  return btoa(binary);
}

function utf8Decode(str: string): string {
  const binary = atob(str);                       // → Latin-1 字符串
  const bytes = new Uint8Array(binary.length);
  for (let i = 0; i < binary.length; i++) bytes[i] = binary.charCodeAt(i);
  return new TextDecoder().decode(bytes);         // → UTF-8 字符串
}
```

**核心思路**:`btoa` 期望「每个字符 = 1 字节」,所以中间必须经过 Latin-1 字符串这一层桥梁。

### 历史方案对比

```ts
// 方案 A(老):unescape + encodeURIComponent,unescape 已废弃,不推荐
btoa(unescape(encodeURIComponent(str)))

// 方案 B(本项目):TextEncoder,现代浏览器原生,推荐
```

### 可扩展方向

- **URL-safe Base64**:把 `+` `/` `=` 替换成 `-` `_` 空,适合放 URL
- **文件 Base64**:支持上传文件,用 `FileReader.readAsDataURL` 输出 dataURL

## 5. 时间戳转换

📄 文件:[TimestampTool.tsx](../src/pages/TimestampTool.tsx)

### 实时时钟

```ts
useEffect(() => {
  const id = setInterval(() => setNow(Date.now()), 1000);
  return () => clearInterval(id);
}, []);
```

每秒更新一次,卸载时清理定时器。`Date.now()` 返回**毫秒**,除以 1000 就是 Unix 时间戳(秒)。

### 时间戳 → 日期

```ts
new Date(ms)  // ms 是毫秒数
```

如果用户输入的是秒,要先 ×1000:

```ts
const ms = unit === 's' ? n * 1000 : n;
new Date(ms);
```

### 日期字符串 → 时间戳

```ts
const d = new Date('2026-05-12 14:30:00');
d.getTime();  // 毫秒
```

> ⚠️ **跨浏览器陷阱**:`new Date('2026-05-12 14:30:00')` 在 Safari 下会返回 `Invalid Date`!Safari 对 `YYYY-MM-DD HH:mm:ss` 格式不友好。

**解决方案**:把 `-` 替换为 `/`,这样所有浏览器都接受:

```ts
new Date(input.replace(/-/g, '/'));
```

或者更严谨地手动 parse 后用 `new Date(year, month-1, day, ...)` 构造。

### 时区

`Date` 对象在内部存的都是**UTC 毫秒**,显示时按用户本地时区。所以同一个时间戳在北京和纽约显示的字符串不同。本工具用本地时区,符合直觉;如需指定时区,要用 `Intl.DateTimeFormat` 或 `dayjs/luxon`。

### 可扩展方向

- **时区切换**:加下拉菜单选时区
- **相对时间**:「3 小时前」「2 天后」
- **批量转换**:一行一个时间戳,批量输出

## 6. 颜色转换

📄 文件:[ColorTool.tsx](../src/pages/ColorTool.tsx)

### 单一数据源

整个组件**只用 RGB 作为状态源**,HEX 和 HSL 都从 RGB 派生:

```ts
const [rgb, setRgb] = useState({ r: 99, g: 102, b: 241 });
const hex = rgbToHex(rgb);
const hsl = rgbToHsl(rgb);
```

任何输入框修改时,先转回 RGB 再 setState。这样**永远不会出现三个值不同步**的 bug。

### HEX ↔ RGB

HEX 就是 16 进制 RGB:

```
#6366f1
 │ │ │
 R G B
```

```ts
function hexToRgb(hex: string) {
  const m = hex.match(/^#?([0-9a-f]{6}|[0-9a-f]{3})$/i);
  if (!m) return null;
  let h = m[1];
  if (h.length === 3) h = h.split('').map(c => c + c).join('');  // #f0a → ff00aa
  return {
    r: parseInt(h.slice(0, 2), 16),
    g: parseInt(h.slice(2, 4), 16),
    b: parseInt(h.slice(4, 6), 16),
  };
}
```

支持 3 位简写(`#f0a` = `#ff00aa`),这是 CSS 标准。

### RGB ↔ HSL

HSL 是为人类设计的色彩模型,直观调节亮度更方便。转换公式:

**RGB → HSL** 算法(简化版):
```
1. 把 R/G/B 归一化到 [0,1]
2. 取 max 和 min
3. L = (max + min) / 2
4. 若 max == min:  H = 0, S = 0(灰色)
   否则:
     S = L > 0.5 ? d/(2-max-min) : d/(max+min)  其中 d = max - min
     H 的扇区:
       max = R: H = (G-B)/d + (G<B ? 6 : 0)
       max = G: H = (B-R)/d + 2
       max = B: H = (R-G)/d + 4
     H = (H/6) * 360
```

**HSL → RGB** 用的是 **HSL 转 RGB 的色相分段算法**(W3C 推荐),在代码里按 H 落在 0/60/120/180/240/300 哪个 60° 扇区分别处理。

公式来自 W3C CSS Color 规范,广泛验证过,不必自己推导。

### 颜色取色器

`<input type="color">` 是 HTML5 原生组件,各平台调用系统取色器,**不需要任何 JS 库**:

```html
<input type="color" value="#6366f1" onChange={...} />
```

返回值永远是 7 位 HEX(`#rrggbb`),这就是为什么 HEX 是这里的「枢纽」格式。

### 可扩展方向

- **HSV、CMYK、Lab** 等色彩空间
- **配色推荐**:互补色、三角色、邻近色
- **WCAG 对比度计算**:输入两个颜色,输出对比度等级(AA/AAA)
- **图片取色**:上传图片提取主色调(用 Canvas + k-means)

## 部署说明

### 静态部署

```bash
npm run build
# 产物在 dist/,直接传到任意静态服务器
```

由于使用了 `HashRouter`,以下平台**零配置**即可部署:
- GitHub Pages
- Vercel / Netlify(也可用 BrowserRouter,平台自动配 fallback)
- 对象存储(S3 / 腾讯云 COS / 阿里云 OSS)
- 任意 Nginx 子目录

### 子路径部署

如果部署在 `https://example.com/tools/`,改 [vite.config.ts](../vite.config.ts):

```ts
export default defineConfig({
  base: '/tools/',
  // ...
});
```

### 隐私声明

所有功能在浏览器本地完成,**没有任何网络请求发送用户数据**。可以放心处理敏感 JSON、私密图片等。如要在公司内部部署,可以加一行声明在页脚以打消用户顾虑(本项目侧边栏已加)。
